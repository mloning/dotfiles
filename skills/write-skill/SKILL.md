# Write Skill

A skill is a reusable workflow or reference guide for future agents — not a story of how you solved something once. Keep it lean (context is a public good; add only what the agent doesn't already know) and make it work for any agent that loads it.

1. **Decide it's worth a skill.** Write one when the technique is reusable across projects, wasn't obvious, and would help a future agent. Don't for one-offs, well-documented standard practice, or project-specific conventions (those belong in `CLAUDE.md`/`AGENTS.md`). If a linter, regex, or script can enforce it mechanically, automate it instead — skills are for judgment.
2. **Ground it in concrete examples.** Before writing, pin down 2–3 real prompts that should trigger the skill and what good output looks like; work backward from those. If the usage is fuzzy, ask me a couple of sharp questions — not a barrage.
3. **Name and place it.** Lowercase letters, digits, hyphens; verb-led; match the existing families (`write-*`, `review-*`, `*-pr`). The folder name equals the `name`, and the skill lives in `skills/<name>/` — this repo's `skills/` is the single source of truth, copied into each agent's dir (edit the source, never the copies).
4. **Write the frontmatter.** Only `name` and `description`. The description is the sole trigger signal — the body loads only after it fires — so write it in third person, stating what the skill does AND the concrete triggers/symptoms for when to use it ("Use when …"), in words someone would search for. Don't spell out the step-by-step workflow there; agents follow the description and skip the body. Add `disable-model-invocation: true` for explicit-only workflow skills and mirror it in `agents/openai.yaml` (`policy: allow_implicit_invocation: false`); omit it for skills meant to auto-trigger.
5. **Write the body — lean and imperative.** Open with one line on the goal or core principle, then tight numbered steps (the house style). Match direction to fragility: prose for judgment calls, pseudocode or parameterized scripts where a pattern exists, exact scripts where the sequence is fragile and order matters. One excellent, runnable example beats five mediocre ones — no multi-language dilution, no narrative storytelling.
6. **Use progressive disclosure.** Keep SKILL.md to the essentials. Push heavy reference (API docs, schemas, long guides) to `references/`, reusable code to `scripts/`, and output templates to `assets/`, each linked from SKILL.md with a note on when to read it (keep references one level deep). No `README`/`CHANGELOG`/`INSTALL` clutter — only files the agent needs to do the job.
7. **Make it cross-agent and composable.** It must run on Claude Code and Codex alike: keep commands and tool references runtime-agnostic, or branch explicitly ("From Claude: … From Codex: …"). To reuse a sibling skill, name it and — for explicit-only ones, which can't be called with the Skill tool — say "Read its SKILL.md and follow it inline." Never `@`-link a skill; that force-loads it and burns context.
8. **Test before trusting it.** Forward-test with a subagent on a realistic task: pass the raw prompt and artifacts the way a user would, don't reveal it's a test or leak the answer you expect. Stronger still — check the baseline without the skill first, so you know it actually changes behaviour. Watch where it struggles, tighten the wording, and re-test (cross-agent: have the other agent run it too).
9. **Sync.** Run `make link-skills` to propagate the new or edited skill to every agent dir, and remove any stale copies left by a rename or deletion.
